## 목적

이전 자동 리뷰(`/team-pr-review`)에서 지적된 🔴 Must Fix / 🟡 Should Fix 항목이 후속 커밋에서 실제로 해결되었는지 검증합니다.
또한, 마지막 AI 리뷰 이후 push된 새 코드에 대해 **증분 리뷰**를 수행하여 리뷰 사각지대를 방지합니다.
해결 여부와 증분 리뷰 결과를 종합하여, 미해결/신규 이슈가 있으면 리마인드 코멘트를 남기고, 전부 해결되었으면 머지 가능 코멘트를 남깁니다.

## 사용법

```
/pr-review-fu <PR번호 또는 URL>
```

## 지시사항

### Phase 1: 이전 리뷰 코멘트 수집

PR의 기존 리뷰 코멘트를 수집하여 지적 항목 목록을 구성합니다.

**수집:**
```bash
# PR의 모든 리뷰 가져오기
gh api repos/{owner}/{repo}/pulls/{pr}/reviews --paginate

# PR의 모든 인라인 코멘트 가져오기
gh api repos/{owner}/{repo}/pulls/{pr}/comments --paginate
```

**파싱 기준:**
- `🔴 [Must Fix]` 또는 `🟡 Should Fix` 태그가 포함된 코멘트만 추적 대상
- 각 코멘트에서 추출할 정보:
  - **파일 경로** (`path`)
  - **라인 번호** (`line` / `original_line`)
  - **점검 단계** (예: `2-4. 정확성`)
  - **문제점 요약** (`**문제점:**` 이후 텍스트)
  - **심각도** (🔴 / 🟡)
- 이미 `resolved`로 표시된 코멘트는 제외

**이전 리뷰가 없는 경우:**
- 추적 항목 0건으로 처리하고, Phase 2를 스킵한 뒤 Phase 2.5(증분 리뷰)로 직접 진행

**출력:** 추적 대상 항목 목록 구성

```
| # | 심각도 | 파일 | 라인 | 점검 단계 | 문제점 요약 |
|---|--------|------|------|-----------|-------------|
| 1 | 🔴 | OrderService.java | 45 | 2-4 | NPE 위험 |
| 2 | 🟡 | OrderController.java | 12 | 2-3 | Repository 직접 호출 |
```

---

### Phase 2: 후속 변경 분석

이전 리뷰 이후의 커밋에서 해당 파일/라인이 수정되었는지 확인합니다.

> **Phase 1에서 추적 항목이 0건이면 이 Phase를 스킵하고 Phase 2.5로 진행합니다.**

**수집:**
```bash
# 마지막 리뷰 이후 커밋 목록
gh api repos/{owner}/{repo}/pulls/{pr}/commits --paginate

# 최신 diff
gh pr diff <PR>
```

**각 추적 항목별 판정:**

1. **해당 파일이 후속 커밋에서 변경되었는지** 확인
2. 변경되었다면, **지적된 라인 근처(±10줄)가 수정되었는지** 확인
3. 수정 내용이 **지적 사항을 실제로 해결하는지** 코드 레벨에서 판정

**판정 결과:**

| 상태 | 기준 |
|------|------|
| ✅ 해결됨 | 지적된 코드가 수정되어 문제가 해결된 것이 확인됨 |
| ⚠️ 부분 해결 | 수정은 되었으나 지적 사항이 완전히 해결되지 않음 |
| ❌ 미해결 | 해당 파일/라인에 변경 없음, 또는 변경이 지적과 무관 |
| 🔄 코드 이동 | 해당 코드가 다른 위치로 이동됨 — 새 위치에서 재검증 필요 |

---

### Phase 2.5: 증분 리뷰

마지막 AI 리뷰 이후 push된 새 코드에 대해 `/team-pr-review`의 Phase 2(코드 변경 분석)와 동일한 점검 기준으로 리뷰합니다.
이를 통해 초기 리뷰 → APPROVE 이후 push된 새 코드가 리뷰 사각지대에 놓이는 것을 방지합니다.

#### 2.5-1. 증분 diff 범위 결정

```bash
# 마지막 AI 리뷰(github-actions[bot])의 commit_id 조회
gh api repos/{owner}/{repo}/pulls/{pr}/reviews --paginate \
  --jq '[.[] | select(.user.login == "github-actions[bot]" and (.body | test("PR 리뷰 결과|증분 리뷰")))] | last | .commit_id'
```

- **마지막 AI 리뷰가 있는 경우**: 해당 `commit_id`부터 현재 HEAD까지의 증분 diff를 대상으로 합니다.
  ```bash
  git diff {last_review_commit_id}...HEAD
  ```
- **마지막 AI 리뷰가 없는 경우**: PR 전체 diff를 대상으로 합니다.
  ```bash
  gh pr diff <PR>
  ```

> **증분 diff가 비어 있으면** (마지막 리뷰 커밋 = HEAD) 이 Phase를 스킵합니다.

#### 2.5-2. 증분 코드 점검

증분 diff에 포함된 변경 파일에 대해, `/team-pr-review` Phase 2의 코드 변경 분석과 동일한 점검 기준을 적용합니다:

- **2-1. Lint & 하드코딩 점검**: 네이밍 컨벤션, 하드코딩, 구조화된 로깅 등
- **2-2. 보안 점검**: SQL Injection, XSS, 인증/인가, 민감 정보 로그 노출 등
- **2-3. 모듈 간 침범 점검**: 레이어 경계 위반, 모듈 의존성
- **2-4. 비즈니스 로직 점검 — 정확성**: 로직 오류, NPE 위험, 엣지 케이스 누락
- **2-5. 비즈니스 로직 점검 — 성능**: N+1 쿼리, 불필요한 I/O, 대량 데이터
- **2-6. 비즈니스 로직 점검 — 동시성 & 트랜잭션**: @Transactional 범위, 경쟁 조건
- **2-7. API 스펙 호환성 점검**: 필드 삭제/타입 변경, 필수값 추가
- **2-8. 설정 파일 변경 점검**: 프로파일 정합성, 의존성
- **2-9. 테스트 코드 점검**: 테스트 존재, 엣지 케이스 커버

#### 2.5-3. 중복 지적 필터링

이전 리뷰(Phase 1에서 수집)에서 이미 지적된 동일 이슈는 증분 리뷰에서 중복 지적하지 않습니다.

**중복 판정 기준:**
- 동일 파일 + 동일/인접 라인(±5줄) + 동일 점검 단계의 지적은 중복으로 간주
- 이전 리뷰에서 지적되었으나 아직 미해결인 항목은 Phase 2에서 이미 추적 중이므로 증분 리뷰에서 제외

#### 2.5-4. 증분 리뷰 결과 정리

증분 리뷰에서 발견된 새 이슈를 심각도별로 분류합니다:

| 심각도 | 기준 |
|--------|------|
| 🔴 Must Fix | 보안 취약점, NPE/크래시 위험, 데이터 정합성 문제, 승인 차단 사유 |
| 🟡 Should Fix | 컨벤션 위반, 성능 개선, 코드 품질 — 권고 사항 |
| 💡 Suggestion | 개선 제안 — 참고용 |

### Phase 3: 결과 종합 및 액션

Phase 2(지적 추적)와 Phase 2.5(증분 리뷰) 결과를 종합하여 최종 판정합니다.

#### 승인 판정 기준 — 🔴 Must Fix만 차단, 🟡 Should Fix는 권고

**중요: 승인 여부는 오직 🔴 Must Fix 항목의 해결 상태 + 증분 리뷰의 새 🔴 Must Fix 존재 여부로 결정합니다.**
🟡 Should Fix 항목은 미해결/신규여도 승인을 차단하지 않습니다. 권고 사항으로만 코멘트에 포함합니다.

| 조건 | 액션 | Exit Code |
|------|------|-----------|
| 기존 🔴 미해결 OR 증분 🔴 존재 | `REQUEST_CHANGES` + 🔴 인라인 코멘트 + 팀 리뷰어 배정 | `1` |
| 🔴 전부 해결 + 증분 🔴 없음, 기존 🟡 미해결 OR 증분 🟡 존재 | `APPROVE` + 미해결/신규 🟡 권고 코멘트 포함 | `0` |
| 전부 해결 + 증분 이슈 없음 | `APPROVE` + 머지 가능 코멘트 | `0` |
| 추적 대상 없음 + 증분 이슈 없음 | `APPROVE` + 머지 가능 코멘트 | `0` |

#### 인라인 코멘트 규칙

- 증분 리뷰에서 발견된 **🔴 Must Fix만** 인라인 코멘트를 남긴다
- 🟡 Should Fix, 💡 Suggestion은 전체 리뷰 요약에만 포함

#### 리뷰 바디 형식

**Case 1: 이전 지적 있음 + 증분 리뷰 수행 → 🔴 미해결/신규 존재 시**

```
## 🔄 리뷰 피드백 추적 + 증분 리뷰 결과: #<PR번호>

### 📌 이전 지적 추적

| # | 심각도 | 파일 | 문제점 | 상태 |
|---|--------|------|--------|------|
| 1 | 🔴 | OrderService.java:45 | NPE 위험 | ❌ 미해결 |
| 2 | 🟡 | OrderController.java:12 | Repository 직접 호출 | ✅ 해결됨 |

#### 미해결 🔴 Must Fix 항목 (승인 차단)

##### ❌ #1 — `OrderService.java:45` (🔴 Must Fix)
- **원래 지적:** NPE 위험
- **현재 상태:** 해당 라인 변경 없음. null 체크 추가 필요.

### 🔍 증분 리뷰: `a1b2c3d`..`e4f5g6h`

| # | 심각도 | 파일 | 점검 단계 | 문제점 |
|---|--------|------|-----------|--------|
| 3 | 🔴 | PaymentService.java:78 | 2-4 | 0으로 나누기 위험 |
| 4 | 🟡 | NewService.java:30 | 2-1 | 매직 넘버 사용 |

#### 🔴 #3 — `PaymentService.java:78` (2-4. 정확성)
- **문제점:** divisor가 0일 수 있으나 체크 없음
- **이유:** ArithmeticException 발생 가능

#### 🟡 #4 — `NewService.java:30` (2-1. Lint)
- **문제점:** 매직 넘버 사용
- **이유:** 상수 추출 필요

### 판정: ❌ 승인 불가 — 🔴 Must Fix 미해결 1건, 증분 🔴 1건
```

**Case 2: 이전 지적 있음 → 🔴 전부 해결 + 증분 🔴 없음, 🟡 미해결/신규 존재**

```
## 🔄 리뷰 피드백 추적 + 증분 리뷰 결과: #<PR번호>

### 📌 이전 지적 추적

| # | 심각도 | 파일 | 문제점 | 상태 |
|---|--------|------|--------|------|
| 1 | 🔴 | OrderService.java:45 | NPE 위험 | ✅ 해결됨 |
| 2 | 🟡 | OrderController.java:12 | Repository 직접 호출 | ❌ 미해결 |

✅ 모든 🔴 Must Fix 항목 해결 완료

### 🔍 증분 리뷰: `a1b2c3d`..`e4f5g6h`

| # | 심각도 | 파일 | 점검 단계 | 문제점 |
|---|--------|------|-----------|--------|
| 3 | 🟡 | NewService.java:30 | 2-1 | 매직 넘버 사용 |

### ⚠️ 미해결/신규 🟡 Should Fix 항목 (권고, 승인 차단하지 않음)

#### ❌ #2 — `OrderController.java:12` (🟡 Should Fix)
- **원래 지적:** Repository 직접 호출
- **현재 상태:** 해당 라인 변경 없음. 향후 개선을 권장합니다.

#### 🆕 #3 — `NewService.java:30` (🟡 2-1. Lint)
- **문제점:** 매직 넘버 사용
- **이유:** 상수 추출 필요

### 판정: ✅ 승인 — 🔴 전부 해결, 증분 🔴 없음 (🟡 미해결 1건 + 증분 🟡 1건은 권고)
```

**Case 3: 전부 해결 + 증분 이슈 없음 → 승인**

```
## 🔄 리뷰 피드백 추적 + 증분 리뷰 결과: #<PR번호>

### 📌 이전 지적 추적

| # | 심각도 | 파일 | 문제점 | 상태 |

### 🔍 증분 리뷰: `a1b2c3d`..`e4f5g6h`

증분 변경에서 새로운 이슈가 발견되지 않았습니다.

```

**Case 4: 이전 리뷰 없음 → 증분 리뷰만 수행**

```
## 🔍 증분 리뷰 결과: #<PR번호>

> 이전 AI 리뷰가 없어 증분 리뷰만 수행합니다.

### 🔍 증분 리뷰 (PR 전체 diff 대상)

| # | 심각도 | 파일 | 점검 단계 | 문제점 |
|---|--------|------|-----------|--------|
| 1 | 🔴 | PaymentService.java:78 | 2-4 | 0으로 나누기 위험 |
| 2 | 🟡 | NewService.java:30 | 2-1 | 매직 넘버 사용 |

#### 🔴 #1 — `PaymentService.java:78` (2-4. 정확성)
- **문제점:** divisor가 0일 수 있으나 체크 없음
- **이유:** ArithmeticException 발생 가능

#### 🟡 #2 — `NewService.java:30` (2-1. Lint)
- **문제점:** 매직 넘버 사용
- **이유:** 상수 추출 필요

### 판정: ❌ 승인 불가 — 증분 🔴 1건
```

이전 리뷰 없고 증분 이슈도 없는 경우:

```
## 🔍 증분 리뷰 결과: #<PR번호>

> 이전 AI 리뷰가 없어 증분 리뷰만 수행합니다.

### 🔍 증분 리뷰 (PR 전체 diff 대상)

증분 변경에서 새로운 이슈가 발견되지 않았습니다.

### 판정: ✅ 승인 — 증분 리뷰 통과
```

#### 실행 명령

> **중요:** `gh pr review --body`에 전달하는 리뷰 본문은 반드시 **HEREDOC**(`cat <<'EOF'`)을 사용하여 실제 줄바꿈이 보존되도록 해야 합니다. 큰따옴표 문자열 안에 `\n`을 넣으면 GitHub에서 마크다운이 렌더링되지 않고 한 줄로 붙어서 표시됩니다.

**🔴 미해결 또는 증분 🔴 존재 시 (Exit Code 1):**

증분 리뷰에서 발견된 🔴 Must Fix 항목은 인라인 코멘트를 남깁니다:
```bash
# 증분 🔴 항목에 대해 인라인 코멘트 (pending review로 묶어서 제출)
gh api repos/{owner}/{repo}/pulls/{pr}/comments -f body="..." -f path="..." -F line=... -f commit_id="..."
```

리뷰 제출 (HEREDOC 필수):
```bash
gh pr review <PR> --request-changes --body "$(cat <<'EOF'
## 🔄 리뷰 피드백 추적 + 증분 리뷰 결과: #<PR번호>

... (위 형식에 따라 작성)
EOF
)"
```

팀 리뷰어 배정:
```bash
gh pr edit <PR> --add-reviewer portlogics-dev/backend
```

마지막에 반드시 출력:
```
EXIT_STATUS: UNRESOLVED
```
그 후 `exit 1`로 종료.

**🔴 전부 해결 + 증분 🔴 없음 시 (Exit Code 0) — 🟡 미해결/신규 여부 무관:**
```bash
gh pr review <PR> --approve --body "$(cat <<'EOF'
## 🔄 리뷰 피드백 추적 + 증분 리뷰 결과: #<PR번호>

... (위 형식에 따라 작성)
EOF
)"
```
마지막에 반드시 출력:
```
EXIT_STATUS: APPROVED
```
정상 종료 (exit 0).

## Exit Code 규칙

GitHub Actions 워크플로우에서 Slack 알림 분기를 결정합니다. **Exit code는 🔴 Must Fix 해결 상태 + 증분 🔴 존재 여부만 반영합니다.**

| 조건 | Exit Code | GitHub 액션 | Slack 알림 |
|------|-----------|-------------|------------|
| 🔴 전부 해결 + 증분 🔴 없음 → PR approved | `0` | `--approve` | Approved (초록) |
| 🔴 미해결 또는 증분 🔴 존재 → PR not approved | `1` | `--request-changes` | Changes Requested (빨강) |
| 증분 리뷰만 수행, 이슈 없음 → PR approved | `0` | `--approve` | Approved (초록) |
| 증분 리뷰만 수행, 🟡만 존재 → PR approved | `0` | `--approve` | Review Posted (노랑) |

## 주의사항

- 이전 리뷰 코멘트가 없어도 증분 리뷰를 실행하여 새 코드를 점검
- 증분 리뷰에서 이전에 이미 지적된 동일 이슈는 중복 지적하지 않음
- 코드 리팩토링으로 파일명/클래스명이 변경된 경우, git rename 감지를 통해 추적
- 판정에 확신이 없으면 ⚠️ 부분 해결로 분류하고, 사람 리뷰어에게 확인 요청
- 모든 코멘트는 한국어로 작성
- 아첨/칭찬 금지 — 객관적 사실만 서술
