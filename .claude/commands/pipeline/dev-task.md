# dev-task — Grill-me + 개발 계획서 작성

⛔ **prod 브랜치 절대 접근 금지**

Context Pack의 태스크 스펙을 기반으로 Grill-me 대화를 진행하고 개발 계획서를 작성합니다.

## 절대 규칙 (반드시 준수)

### 질문 규칙
- 한 번에 하나의 질문만. 사용자 답변 후 다음 질문.
- **코드로 알 수 있는 건 묻지 마세요.** 필드 타입, 엔티티 관계, 메서드 시그니처 등은 코드에서 직접 확인. 사용자에게는 비즈니스 판단만 질문.
- **기술적 구현 방식은 혼자 판단하세요.** 매퍼 시그니처, DTO 패턴, 쿼리 방식 등 기존 코드 패턴을 따르면 되는 건 묻지 않고 처리.
- **"답변이 되었을까요?" 금지.** 사용자가 답변하면 "알겠습니다." 하고 다음 질문으로 바로 넘어가세요. 불필요한 확인 질문 하지 마세요.

### 코드베이스 규칙
- **코드베이스 스캔은 Step 0에서 1회만.** Grill-me 대화 중 추가 스캔 금지.
- **현재 작업 브랜치({TASK_ID})만 사용.** 다른 브랜치 참조/탐색 금지.

### 언어
- 한국어로만 대화합니다.

### 시작 메시지
가장 먼저 아래 형식으로 출력:
📋 **{TASK_ID}** — {TASK_NAME}
파이프라인을 시작합니다.

## Arguments
- `$ARGUMENTS` — 업무번호(BE-XXXX)와 태스크 설명

## Execution

### Step 0-0: 프로젝트 컨텍스트 로드 (최우선)

프로젝트 루트에 `.cortx/project-context.md`가 존재하면 **가장 먼저 Read**합니다. 이 파일은 cortx 스캐너가 생성한 프로젝트 규칙 인덱스로, 불필요한 탐색을 건너뛰게 해줍니다.

- `## Scan Quality` 섹션의 Overall 등급 확인:
  - `rich`: 이어지는 `Task-Type → Doc Mapping`, `Immediate Rules`, `Common Pitfalls`를 그대로 신뢰
  - `partial`: 누락된 항목은 사용자에게 확인
  - `sparse` 또는 SOT가 `referenced_but_empty`: **첫 질문으로** "규칙 문서가 비어있습니다. 먼저 CLAUDE.md/SOT를 작성하시겠습니까, 아니면 그대로 진행할까요?" 질문
- `## Task-Type → Doc Mapping` 표가 있으면, 현재 태스크 유형에 해당하는 행의 문서를 **우선 Read** (예: 구현 태스크 → `.ai/docs/architecture.md`)
- `## Immediate Rules` 블록은 **즉시 적용** (추가 문서 로드 없이)
- 파일이 없으면 아래 Step 0으로 진행 (fallback)

### Step 0: 컨텍스트 확인

[PIPELINE:grill_me:in_progress]

Context Pack에 제공된 내용(Notion 페이지, Slack 메시지 등)을 읽고 태스크 스펙을 파악합니다.
- 스펙에 오기/모순이 있을 수 있으므로 코드베이스와 교차 검증 필수
- Context Pack에 충분한 정보가 없으면 사용자에게 보충을 요청

### Step 1: Grill-me 대화

코드베이스 탐색 결과를 **반드시 아래 형식으로** 정리한 후 첫 질문을 합니다:

```
## Grill-me 시작 — {TASK_ID} {TASK_NAME}

### 현재 구조 파악
- **대상 API**: `{엔드포인트}`
- **컨트롤러**: `{클래스명}` ({모듈})
- **서비스**: `{클래스명}` — {설명}
- **리포지토리**: `{클래스명}` — {설명}
- **연관관계**: `{엔티티}` → `{엔티티}` ({관계 타입})

### Notion 스펙 요약
- {스펙 핵심 1}
- {스펙 핵심 2}
- {스펙 핵심 3}

### 주요 이슈/병목
| 위치 | 패턴 | 심각도 |
|------|------|--------|
| `{클래스.메서드()}` | {설명} | {Critical/High/Medium} |

---

**Q1.** {비즈니스 판단이 필요한 질문}?
```

**중요**: 질문 형식은 반드시 `**Q1.**`처럼 마크다운 bold(`**`)로 감싸야 합니다. 앱이 이 형식을 감지해 사용자에게 Asking 상태로 표시합니다.

**규칙**:
1. 모든 질문과 대화는 **한국어**로 진행
2. **한 번에 하나의 질문**만 합니다
3. **project-context.md가 이미 답하는 것은 Read하지 말 것.** Tech Stack, 모듈 목록, SOT, Immediate Rules, Forbidden Paths는 이미 로드됨 — 재확인 금지. 코드를 실제로 읽어야만 답할 수 있는 구체적 지점(특정 메서드 시그니처, 엔티티 필드 등)만 Read.
4. 코드베이스에서 확인/추측한 결과를 먼저 제시하고, 비즈니스 의도/판단만 질문
5. 도메인 문서는 도메인 비즈니스 태스크에서만 참조
6. **질문 형식은 반드시 `**Q1.**`, `**Q2.**`, `**Q3.**` ... 으로 번호를 매겨서(마크다운 bold) 합니다.** 질문은 항상 `?`로 끝냅니다.
7. **첫 질문까지 1분 이내 도달 목표** — 과도한 코드 탐색 금지. project-context.md + Context Pack으로 파악한 구조를 바탕으로 빠르게 첫 `**Q1.**` 출력.

**질문 영역**: 요구사항 명확화 → 기술적 제약 → 설계 결정 → 엣지 케이스 → 테스트 전략

"끝", "완료", "다음", "done" 입력 시 Step 2로 진행.
